# Prompt 工程实战：从新手到老手

我用 AI 写代码大概一年多了，从最开始的"帮我写个 XX"到现在能用 Prompt 精确控制输出质量，中间经历了好几个阶段。这篇就分享一下我摸索出来的 Prompt 技巧。

## 第一阶段：把 AI 当搜索引擎用

最开始我这样写 Prompt：

> 帮我写一个防抖函数

AI 确实能写，但出来的东西要么太简单要么不适用。后来发现问法决定了答案质量。

## 第二阶段：给上下文

更好的问法是这样的：

```
我们的项目用 Vue 3 + TypeScript，现在有个搜索输入框，
需要在用户停止输入 500ms 后发起 API 请求。
请帮我写一个可复用的 useDebounce hook，
要求：
1. 支持泛型
2. 返回 debounced 值和取消函数
3. 组件卸载时自动清理
```

区别在哪？**给了上下文和具体要求**。AI 不需要猜你要什么，直接把精准的东西给你。

## 第三阶段：角色 + 格式 + 约束

这是我现在的写法：

```
你是我的技术合伙人，负责代码审查。
请 review 以下代码，按这个格式输出：

## 问题
- 位置：[文件名:行号]
- 严重程度：[高/中/低]
- 说明：为什么这是个问题
- 建议：如何修改

约束：
- 只说真问题，别说代码风格偏好
- 如果没问题就说"没问题"
- 优先关注安全和性能问题
```

效果对比：

| 方式 | 有效信息 | 废话 |
|------|---------|------|
| 无格式 | 30% | 70% |
| 有格式 | 85% | 15% |

## 写 Prompt 的四个要素

我总结了一个叫 **RICE** 的框架：

### R — Role（角色）
告诉 AI 它应该扮演什么角色。"你是一个资深前端工程师"比直接提问好得多。

### I — Intent（意图）
你要做什么，为什么做。AI 理解意图后能给出更符合场景的方案。

### C — Context（上下文）
相关的背景信息：技术栈、现有代码、团队规范等。

### E — Expectation（期望）
你期望什么样的输出：格式、长度、角度等。

## 实战示例

### 差 Prompt
> 帮我写个登录页面

（太模糊，AI 只能输出一个最基本的 demo）

### 好 Prompt
```
Role: 你是一个有 5 年经验的 React 开发工程师
Intent: 我们需要一个用户登录页面，作为管理后台的入口
Context: 
- 技术栈：React 18 + TypeScript + Ant Design
- 已有 API：POST /api/auth/login，接受 {username, password}，返回 {token}
- 已有 auth store：useAuthStore，有 login 方法
Expectation:
- 处理 loading、error、表单验证状态
- 错误信息在页面上显示，不用弹窗
- 登录成功后跳转到 /dashboard
- 代码加上关键注释，不用解释为什么
```

## 调 Prompt 的技巧

1. **第一次不完美很正常** — 迭代式调整比一次写对更容易
2. **告诉 AI 哪里错了** — "你输出的代码里缺少错误处理，请补充"比重新问一次有效率
3. **用例子说明** — "就像这个文件里的写法一样" + 贴一段代码
4. **限制输出长度** — "用三句话概括"、"只说关键点"
5. **用对比引导** — "方案 A 和方案 B 哪个更好？为什么？"

## 踩过的坑

### 过度信任 AI
AI 写出来的代码有时候看起来很有道理，但实际上有逻辑错误。尤其是复杂的业务逻辑，一定要自己理解后再用。

### 上下文丢失
长对话里 AI 会忘记之前的约定。比如前面说了"用 React"，后面它可能就给你写 Vue 了。关键约束要在每条 Prompt 里重复。

### 过于抽象的指令
"优化这段代码"这种太抽象了。具体说"把这段代码的时间复杂度从 O(n²) 降到 O(n)"效果更好。

## 写在最后

Prompt 工程不是什么玄学，核心就一句话：**把你希望 AI 知道的、用清晰的结构告诉它**。你对问题理解得越清楚，AI 给你的答案就越有用。
